## Задача 3

1. 100 ВМ, Windows-based, балансировка, репликация, бэкапы
Выбор: VMware vSphere Suite (vCenter + ESXi)

Windows-based инфраструктура	Наилучшая поддержка Windows гостевых ОС, интеграция с Active Directory, инструменты VMware Tools для Windows.
Программные балансировщики	NSX-T (или NSX-v) — продвинутые функции балансировки, микросегментация, overlay-сети.
Репликация данных	vSphere Replication — встроенная репликация ВМ на другой сайт.
Автоматизированные бэкапы	Veeam Backup & Replication — отраслевой стандарт для VMware, идеально интегрируется.
Управление 100+ ВМ	vCenter Server — централизованное управление, шаблоны, кластеризация, DRS, HA.
Общие задачи, нет особых требований	vSphere — наиболее предсказуемая, стабильная платформа с огромной экосистемой.


2. Наиболее производительное бесплатное open-source решение для 20-30 ВМ
Выбор: Proxmox VE (или oVirt)
Бесплатное open-source	Основан на Debian + KVM/LXC, все функции free (кроме enterprise-репозитория).
Производительность	Использует KVM с аппаратной виртуализацией (VT-x/AMD-V) + возможность паравиртуализации (virtio). Практически «железная» производительность.
Управление 20-30 ВМ	Единый web-интерфейс, кластеризация, live migration.
Поддержка Linux и Windows	KVM отлично работает с обоими (для Windows — драйверы virtio).
Интеграция	Встроенная поддержка Ceph, ZFS, бэкапы, репликация.

Proxmox VE — лучший баланс производительности, функциональности и простоты для небольших инфраструктур.

3. Бесплатное, совместимое и производительное решение для Windows-инфраструктуры
Выбор: Hyper-V Server (бесплатный) + Windows Admin Center

Максимальная совместимость с Windows	Один вендор, единые драйверы, Integration Services встроены, идеальная поддержка гостевых Windows.
Бесплатное	Hyper-V Server — free standalone продукт (только роль гипервизора).
Производительность	Прямой доступ к железу, минимальные накладные расходы, поддерживает GPU-PV, SR-IOV.
Управление	Windows Admin Center (бесплатно) или RSAT (Hyper-V Manager) для базового управления.
Стабильность драйверов




4. Окружение для тестирования ПО на нескольких дистрибутивах Linux
Выбор: Vagrant + VirtualBox/Libvirt + Ansible (или Docker)

Тестирование на нескольких дистрибутивах	Vagrant-файлы с описанием боксов (Ubuntu, CentOS, Fedora и т.д.) — поднимаются одной командой.
Рабочее окружение	Изоляция от хоста, возможность «ломать» и пересоздавать ВМ за секунды.
Автоматизация настройки	Провиженинг через Ansible, Puppet, Chef или shell-скрипты прямо в Vagrantfile.
Бесплатность	Весь стек open-source.
Переносимость	Vagrantfile можно передать команде, всё развернётся идентично.
Vagrant — стандарт де-факто для тестовых сред разработки, идеально подходит для сценария «мне нужно быстро развернуть 5 разных Linux, настроить их и тестировать».

## Задача 4

#### Проблемы и недостатки гетерогенной среды виртуализации
1. Операционные сложности
Разные интерфейсы управления — персоналу нужно знать несколько систем (vSphere, Hyper-V, KVM и т.д.).

Непоследовательные процессы — резервное копирование, мониторинг, обновления требуют разных подходов.

Сложность автоматизации — нужны отдельные скрипты/инструменты для каждой платформы.

2. Экономические издержки
Лицензионные затраты — оплата нескольких коммерческих систем (VMware + Microsoft + другие).

Поддержка и обучение — дополнительные расходы на сертификации персонала.

Неэффективное использование ресурсов — нельзя перераспределить ресурсы между разными гипервизорами.

3. Технические риски
Отсутствие единого мониторинга — сложно получить целостную картину инфраструктуры.

Проблемы совместимости — разные форматы ВМ (VMDK, VHDX, QCOW2), миграции становятся нетривиальными.

Разная реализация сетей и хранилищ — сложности с организацией единого Data Center fabric.

4. Вопросы безопасности
Увеличение attack surface — каждая система имеет свои уязвимости.

Несогласованные политики безопасности — разные системы управления доступом, аудита.

Сложность соответствия стандартам (PCI DSS, ISO 27001) — требуется отдельная аттестация каждого компонента.

5. Кадровые проблемы
Дефицит специалистов — сложно найти сотрудников, глубоко знающих несколько платформ.

Снижение эффективности — время тратится на переключение между системами, а не на решение задач.
